# Public API rate-limit headers

Customers building integrations against our REST API have no programmatic way to know how close they are to a rate-limit ceiling. Today they discover the limit by hitting a 429 in production and then opening a support ticket, which is a terrible developer experience for a self-serve API.

We should emit standard `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers on every authenticated response, matching the GitHub / Stripe convention. SDK consumers can then back off gracefully and surface clear errors to their own users instead of treating a 429 as an opaque platform failure.

This is a small, well-bounded change with disproportionately good DX leverage. It also unblocks the OpenAPI spec we are about to publish, since the headers should be documented as part of the contract.
